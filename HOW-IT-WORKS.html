How the system works.

The design system lives as one shared memory. Here is how the studio, the client tool, the AI, and the codebase fit together.

01

The model: one shared memory

The model: one shared memory

The design system is stored as one shared memory — a single place where every color, component, rule, screen, and flow lives.

Both the designer and the AI read from and write back to this memory while building, so everyone works from one source of truth.

03

The three parts

Studio, client tool, and the data.

1

The Studio our side

We set up and fine-tune a client's design system from their existing notes — shaping components and the rules for using them — then deliver their own copy.

2

The Client Tool their side

The copy each client runs on their own machine, used three ways: a visual app to see and play with the whole system, a hands-on toolkit for direct or automated work, and a live connection their AI assistant plugs into. Each copy is locked to one product.

3

The Data

Supporting material — brand guides, product specs — sitting alongside the system so the AI has the full picture, not just the parts.

04

What's inside the client tool

A box of parts — with instructions.

Not just a box of parts — a box of parts with instructions.

Design tokens

The core values: colors, spacing, fonts, sizes. Change them once here and they update everywhere in the code.

Atoms to full components

The smallest pieces — a color, a button, an input — up to complete components. Each carries its own usage rules: when to use, when not to, what to use instead, and which tokens it should use. This is what lets the AI choose the right component and token instead of guessing.

Icons & assets

A built-in library for icons, logos, and images — like a lightweight content system — so the AI always pulls the approved asset.

It's a box of parts with instructions — so both people and AI know exactly what to use and when.

05

How the AI plugs in

A live link, not a stale copy.

The client's AI connects through a live link — an MCP connection, the standard way AI assistants use outside tools. Through it, while it builds, the AI can:

LOOK THINGS UP

"Which component fits here?" "What's the right spacing value?"

GET ITS PLAN CHECKED

The tool flags anything that breaks the design rules before a line of code is written.

WRITE BACK

Anything new the AI builds is saved into the shared memory for next time.

uses approved Button + spacing token
off-token color → flagged before code

Because the link is live, the AI always works from the current system, not a stale copy. For simpler cases the system is also exported as plain files the AI can read — the live link is what powers the real back-and-forth.

06

How it works in practice

From setup to a self-enriching loop.

  1. We set up a client's design system in the Studio and hand them their own copy.
  2. They open the visual app to explore the system and plan a feature — sketching screens and flows, picking components.
  3. The tool bundles exactly what's relevant and hands it to the client's AI.
  4. The AI builds the feature — checking each choice against the rules as it goes — and files anything new back into the shared memory.
  5. The updated system is automatically written into their codebase as ready-to-use design files.

A loop: every feature makes the system richer.

07

Going further: architecture & flows

A richer layer on top of the essentials.

Beyond the design system itself, the tool can map the product — its screens (the information architecture) and how people move between them (user flows). A designer sketches a new feature's flow visually, and the tool turns it into a precise brief for the AI to build from.

This goes further than most design systems. It's a layer added on top of the essentials rather than a day-one requirement — but it's what makes planning and building a whole feature feel seamless, instead of just dropping in a single component.

Add-on layer

Map the screens, draw the flows, hand the AI a precise brief — so a whole feature, not just one component, comes together coherently.

08

Why it's different

Three things that set it apart.

A living memory, not stale docs

The system updates itself as the product grows, so it never falls out of date.

The AI makes consistent decisions

It doesn't just store components — it knows when to use which, and checks the plan against the rules before any code.

The data stays with the client

It turns a product's design system into a shared memory that keeps a team's AI assistant building consistently — and gets smarter with every feature.
